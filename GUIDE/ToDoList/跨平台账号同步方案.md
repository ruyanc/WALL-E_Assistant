# WALL-E 跨平台账号登录与数据同步方案

> 本文档说明 WALL-E「登录账号 + 多平台数据同步」的设计与实现。  
> **实现状态（2026-06）**：**Windows / macOS / Android** 均已接入 **CloudBase**（`walle/sync/`）。用户操作见 [操作手册.md](../../操作手册.md) 第十六节、[CLOUDBASE_SETUP.md](../sync/CLOUDBASE_SETUP.md)。

---

## 1. 现状

| 平台 | 数据目录 | 同步 |
|------|----------|------|
| Windows | `%APPDATA%\WALL-E` | ✅ CloudBase（授权码 + 手机号） |
| Android | Kivy `user_data_dir` | ✅ 同上 |
| macOS | `~/Library/Application Support/WALL-E` | ✅ 与 Windows 相同（授权码 + 手机号） |

桌面端读写逻辑见 `walle/config.py`、`walle/todo_manager.py`、`walle/notes_manager.py`、`walle/reminder_manager.py`；移动端见 `mobile/storage.py`、`mobile/todo_store.py`。

**结论**：要实现跨平台同步，需要用**云端账号**标识数据归属，用**同步协议**替代各端直接读写本地 JSON。

## 2. 推荐整体架构

```
┌─────────────┐  ┌─────────────┐  ┌─────────────┐
│ WALL-E 桌面  │  │ WALL-E macOS │  │   Android   │
└──────┬──────┘  └──────┬──────┘  └──────┬──────┘
       │                │                │
       └────────────────┼────────────────┘
                        ▼
              ┌──────────────────┐
              │ 本地缓存 (JSON/   │
              │ SQLite) +        │
              │ SyncEngine       │
              └────────┬─────────┘
                       │ 拉取 / 推送 / 冲突合并
                       ▼
              ┌──────────────────┐
              │ 云端：Auth + API  │
              │ + 用户数据存储    │
              └──────────────────┘
```

核心原则：

1. **账号**：登录后获得 `access_token`，请求均带用户身份。
2. **本地优先**：先读写本地，保证离线可用；联网后再同步。
3. **统一数据模型**：待办 / 记事 / 提醒使用全局唯一 `id`，并附带 `updated_at`、`deleted`（软删除）。
4. **共享同步逻辑**：抽成 `walle/sync/` 包，桌面与 mobile 复用。

## 4. 后端方案对比

| 方案 | 适合场景 | 优点 | 缺点 |
|------|----------|------|------|
| **Supabase**（推荐起步） | 个人 / 小团队 | 自带邮箱与第三方登录、PostgreSQL、REST 开箱即用 | 需考虑国内访问 |
| **Firebase** | 快速上线 | Auth + Firestore 实时同步 | 国内不稳定；数据模型偏 NoSQL |
| **LeanCloud / 腾讯云 CloudBase** | 国内用户为主 | 国内访问较好；CloudBase 有免费体验环境 | 绑定平台；Python 客户端需走 HTTP API |
| **自建 FastAPI + PostgreSQL** | 长期可控 | 完全自主 | 运维与开发量最大 |

个人项目建议从 **Supabase**（海外访问可接受）或 **腾讯云 CloudBase**（国内为主）起步，不必一开始就自建服务器。CloudBase 具体接入步骤见 [4.1 节](#41-腾讯云-cloudbase-接入指南国内用户推荐)。

#### 4.1.2 环境创建与控制台配置

1. 登录 [腾讯云 CloudBase 控制台](https://console.cloud.tencent.com/tcb)，创建 **免费体验版** 环境，记录 **环境 ID**（形如 `cloud1-xxx`）。
2. **地域**：身份认证相关能力目前仅支持 **上海（ap-shanghai）**，创建时优先选上海。
3. **身份认证** → 登录授权：
   - 开启 **邮箱登录** 或 **用户名密码登录**；

4. **文档型数据库** → 新建集合 `sync_records`（见 4.1.3）。
5. **安全规则** → 为 `sync_records` 配置「仅登录用户读写自己的数据」（见 4.1.4）。
6. **云函数** → 新建 `syncPull`、`syncPush`（见 4.1.6），并在 **HTTP 访问服务** 中为二者开启路由与 **身份验证**。

```python
ENV_ID = "cloud1-xxxxxxxx"  # 环境 ID
API_BASE = f"https://{ENV_ID}.api.tcloudbasegateway.com"
SYNC_PULL_URL = f"{API_BASE}/v1/functions/syncPull"
SYNC_PUSH_URL = f"{API_BASE}/v1/functions/syncPush"
```

#### 4.1.3 数据库集合设计

采用 **单集合存全部同步记录**，便于统一按 `updated_at` 增量查询，与本文第 5 节 LWW 协议一致。

**集合名**：`sync_records`

**文档 `_id` 建议**：`{type}_{record_id}`，例如 `todo_a1b2c3d4-...`，便于 upsert。

| 字段 | 类型 | 说明 |
|------|------|------|
| `_openid` | string | CloudBase 自动注入的登录用户标识，安全规则依赖此字段 |
| `record_id` | string | 业务 UUID，与本地 `id` 一致 |
| `type` | string | `todo` / `note` / `reminder` / `setting` |
| `payload` | object | 该条记录的完整业务字段（`text`、`done`、`priority` 等） |
| `updated_at` | number | Unix 时间戳（秒），LWW 比较依据 |
| `deleted` | bool | 软删除标记 |

**索引**（控制台 → 数据库 → 索引管理）：

- 复合索引：`(_openid, updated_at)` —— 拉取增量时使用
- 可选：`(_openid, type, updated_at)` —— 按类型分批同步

**设置类数据**（番茄钟、`language` 等）可存为 `type: "setting"`、`record_id: "pomodoro"` 等固定键，整条记录 LWW 覆盖。

#### 4.1.4 安全规则示例

在 CloudBase 控制台 → 数据库 → `sync_records` → 权限设置，使用自定义安全规则：

```json
{
  "read": "auth != null && doc._openid == auth.openid",
  "write": "auth != null && (doc._openid == auth.openid || doc._openid == null)"
}
```

说明：云函数内使用服务端 SDK 时可绕过部分限制；若同步逻辑 **全部放在云函数** 中执行，客户端不直连数据库，则客户端只需携带 `access_token` 调用云函数，数据库规则由函数内校验 `auth.openid`，更安全。

#### 4.1.5 认证：HTTP 登录（Python 示例）

WALL-E 客户端登录后，将 `access_token`、`refresh_token`、`expires_at`、`sub`（用户 ID）写入本地 `auth.json`；**密码不落盘**。

```python
# walle/sync/cloudbase_auth.py（概念示意）
import time
import httpx

def sign_in(email: str, password: str, env_id: str) -> dict:
    base = f"https://{env_id}.api.tcloudbasegateway.com"
    resp = httpx.post(
        f"{base}/auth/v1/signin",
        json={"username": email, "password": password},
        timeout=30,
    )
    resp.raise_for_status()
    data = resp.json()
    return {
        "user_id": data["sub"],
        "access_token": data["access_token"],
        "refresh_token": data["refresh_token"],
        "expires_at": time.time() + data.get("expires_in", 7200),
    }

def refresh_token(refresh_token: str, env_id: str) -> dict:
    base = f"https://{env_id}.api.tcloudbasegateway.com"
    resp = httpx.post(
        f"{base}/auth/v1/token/refresh",
        json={"refresh_token": refresh_token},
        timeout=30,
    )
    resp.raise_for_status()
    data = resp.json()
    return {
        "access_token": data["access_token"],
        "refresh_token": data.get("refresh_token", refresh_token),
        "expires_at": time.time() + data.get("expires_in", 7200),
    }
```

要点：

- `access_token` 默认有效期 **2 小时**，SyncEngine 调用前应检查 `expires_at`，过期则用 `refresh_token` 刷新（refresh 有效期约 30 天）。
- 邮箱 **首次注册** 需用户点击验证邮件激活账号，之后方可密码登录。
- 请求云函数时在 Header 携带：`Authorization: Bearer {access_token}`。

官方参考：[用户登录 HTTP API](https://docs.cloudbase.net/http-api/auth/auth-sign-in)、[AccessToken 说明](https://docs.cloudbase.net/http-api/basic/access-token)。

#### 4.1.6 同步 API：云函数方案（推荐）

Python 客户端 **不直连** 文档库复杂查询，由两个云函数封装增量同步，与第 5.2 节流程一一对应。

**syncPull**（拉取云端变更）

- 入参：`{ "last_sync_at": 1718200000 }`
- 逻辑：`db.collection('sync_records').where({ _openid: auth.openid, updated_at: _.gt(last_sync_at) }).limit(1000).get()`
- 出参：`{ "records": [ ... ], "server_time": 1718209999 }`

**syncPush**（推送本地变更）

- 入参：`{ "records": [ { "record_id", "type", "payload", "updated_at", "deleted" }, ... ] }`
- 逻辑：对每条记录按 `_id = type + '_' + record_id` 做 **set/upsert**；若云端已有同 `record_id` 且 `updated_at` 更新，则拒绝覆盖（服务端 LWW 双保险）。
- 出参：`{ "accepted": 12, "rejected": 0 }`

云函数（Node.js）骨架示意：

```javascript
// syncPull/index.js
const cloudbase = require("@cloudbase/node-sdk");
exports.main = async (event, context) => {
  const app = cloudbase.init({ env: context.TCB_ENV });
  const db = app.database();
  const _ = db.command;
  const { last_sync_at = 0 } = event;
  const { openid } = app.auth().getUserInfo(); // 来自 access_token
  const res = await db.collection("sync_records")
    .where({ _openid: openid, updated_at: _.gt(last_sync_at) })
    .limit(1000)
    .get();
  return { records: res.data, server_time: Math.floor(Date.now() / 1000) };
};
```

**客户端调用示例**（与 `SyncEngine` 对接）：

```python
def pull_changes(session, last_sync_at: int) -> list:
    resp = httpx.post(
        SYNC_PULL_URL,
        headers={"Authorization": f"Bearer {session.access_token}"},
        json={"last_sync_at": last_sync_at},
        timeout=60,
    )
    resp.raise_for_status()
    return resp.json()["records"]

def push_changes(session, records: list) -> dict:
    resp = httpx.post(
        SYNC_PUSH_URL,
        headers={"Authorization": f"Bearer {session.access_token}"},
        json={"records": records},
        timeout=60,
    )
    resp.raise_for_status()
    return resp.json()
```

在控制台为上述云函数配置 **HTTP 访问路径** 并 **开启身份验证**，否则会出现 `MISSING_CREDENTIALS` 错误。

#### 4.1.7 与 `walle/sync/` 模块映射

| 方案文档概念 | CloudBase 对应 |
|--------------|----------------|
| `AuthSession` | `auth.json` 存 token；`user_id` = HTTP 登录返回的 `sub` |
| `SyncedStore` | 本地仍读写 JSON；变更后由 SyncEngine 调 `syncPush` |
| `SyncEngine` 拉取 | 调 `syncPull` → 按 `record_id` + `updated_at` 合并到本地 Manager |
| `SyncEngine` 推送 | 收集本地 `updated_at > last_sync_at` 的记录 → 调 `syncPush` |
| `last_sync_at` | 本地 `%APPDATA%/WALL-E/sync_state.json` 持久化 |

未登录时行为不变，仍走 `LocalStore`（见第 6.1 节）。

## 5. 同步协议设计

不要整文件覆盖上传，否则多设备同时修改会丢数据。推荐 **按记录增量同步 + 最后写入胜出（LWW，Last-Write-Wins）**。

### 5.1 记录结构示例

```json
{
  "id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "type": "todo",
  "text": "交周报",
  "done": false,
  "priority": 2,
  "updated_at": 1718200000,
  "deleted": false
}
```

### 5.2 同步流程

1. 登录 → 获取 `user_id` + `access_token`
2. 客户端上报 `last_sync_at`
3. 拉取云端 `updated_at > last_sync_at` 的变更
4. 合并到本地（同 `id` 比较 `updated_at`，较新者覆盖）
5. 推送本地 `updated_at > last_sync_at` 的变更
6. 更新 `last_sync_at`

冲突处理可先采用 **LWW**；若需更精细，再升级为 CRDT 或「冲突时由用户选择」。

## 6. 代码改造步骤

### 6.1 抽象存储层（不动 UI）

将各 Manager 的「读文件 / 写文件」抽成接口：

```python
# walle/sync/store.py（概念示意）
class DataStore(Protocol):
    def load_todos(self) -> dict: ...
    def save_todos(self, data: dict) -> None: ...

class LocalStore(DataStore):
    """现有 JSON 逻辑"""

class SyncedStore(DataStore):
    """本地 + 云端"""
```

- 未登录：走 `LocalStore`
- 已登录：走 `SyncedStore`

### 6.2 认证模块

```python
# walle/sync/auth.py
class AuthSession:
    user_id: str
    access_token: str
    expires_at: float
```

- 桌面（PyQt）：在控制台增加「账号」页（登录 / 登出 / 同步状态）
- Android（Kivy）：在 `mobile/main.py` 增加同等入口
- Token 存于 `%APPDATA%/WALL-E/auth.json`（或各平台等价路径），**勿明文存密码**

### 6.3 SyncEngine

建议触发时机：

- 应用启动且已登录
- 用户点击「立即同步」
- 数据变更后 debounce 3–5 秒自动推送
- 定时后台同步（如每 15 分钟）

### 6.4 统一 ID（重要）

当前 `next_id` 为自增整数，多设备会冲突。同步上线前应改为 **UUID**：

```python
# 改前
Task(id=1, text="开会", ...)

# 改后
Task(id="a1b2c3d4-...", text="开会", updated_at=..., ...)
```

可做一次性迁移：旧整数 `id` 在首次同步时映射为 UUID。

### 6.5 首次登录合并策略

| 情况 | 处理 |
|------|------|
| 本地有数据、云端为空 | 全量上传 |
| 云端有数据、本地也有 | 按 `updated_at` 合并 |
| 任意情况 | UI 提示：「登录后将与云端数据合并」 |

## 7. 各平台注意点

### 7.1 桌面（Windows / macOS）

- HTTP 客户端：`requests` 或 `httpx`
- 提醒调度仍用本地 `ReminderManager`，仅同步提醒**规则**数据
- 宠物窗口相关配置保持本地

### 7.2 Android

- 与桌面共用 `walle/sync/`（buildozer 将该包打入 APK）
- 需先补齐 mobile 的 notes、reminders，再实现完整同步
- 后台同步受 Android 省电策略限制，需考虑前台服务或 WorkManager

### 7.3 安全

- 仅传 token，不传密码
- 必须使用 HTTPS
- 敏感字段端到端加密可作为后期增强
