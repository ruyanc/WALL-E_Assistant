# WALL-E 跨账号任务派发方案

> 本文档说明：在「登录账号 + 跨平台同步」基础上，如何实现**不同账号之间互相派发任务**。  
> 前置阅读：[跨平台账号同步方案.md](./跨平台账号同步方案.md)

当前版本（v1.1.0）的 `Task` 仅为个人本地待办（`walle/todo_manager.py`），无 `owner`、`assignee` 等字段，也无可跨用户访问的服务端。**任务派发必须依赖云端账号体系与协作 API**。

---

## 1. 与「个人同步」的区别

| 能力 | 个人同步 | 跨账号任务派发 |
|------|----------|----------------|
| 数据归属 | 仅当前登录用户 | 涉及**派发者**与**接收者**两个（或多个）用户 |
| 读写权限 | 本人可读可写 | 接收者只能改约定字段；派发者可撤回、查看进度 |
| 冲突处理 | 同用户多设备 LWW | 多用户协作 + 状态机，不能简单覆盖 |
| 通知 | 可选 | **必需**（新任务、被接受、已完成等） |
| 本地模型 | 单份 `todos.json` | 需区分「我的待办」与「他人派给我的」 |

**结论**：派发不是同步的延伸，而是**协作域（Collaboration）**；需在云端维护任务关系与权限，客户端拉取后映射到本地 UI。

---

## 2. 推荐整体架构

```
┌─────────── 账号 A（派发者）───────────┐     ┌─────────── 账号 B（接收者）───────────┐
│  WALL-E 客户端                        │     │  WALL-E 客户端                        │
│  ├─ 我的待办（personal）              │     │  ├─ 我的待办（personal）              │
│  └─ 我派出的任务（outbox）            │     │  └─ 派给我的任务（inbox）             │
└──────────────────┬────────────────────┘     └──────────────────┬────────────────────┘
                   │                                              │
                   └──────────────────┬───────────────────────────┘
                                      ▼
                        ┌─────────────────────────────┐
                        │ 云端协作服务                 │
                        │ ├─ Auth（用户身份）          │
                        │ ├─ User Directory（查人）    │
                        │ ├─ TaskAssignment API       │
                        │ ├─ Notification / Push      │
                        │ └─ PostgreSQL / 文档库       │
                        └─────────────────────────────┘
```

设计原则：

1. **云端为协作真相源**：派发、接受、完成等状态以服务端为准。
2. **个人待办仍可本地优先**：仅「协作任务」走派发通道；个人 `Task` 可继续沿用同步方案。
3. **最小权限**：接收者默认不能改派发者私人字段（如原始备注中的敏感信息可拆分）。
4. **可审计**：保留 `created_by`、`assigned_to`、状态变更历史（至少服务端日志）。

## 3. 核心概念

| 概念 | 说明 |
|------|------|
| **Personal Task** | 用户自建、仅本人可见的待办（现有 `Task`） |
| **Assigned Task** | 由 A 创建并指定 B 执行的任务副本 |
| **Assignment** | 派发关系记录：谁派给谁、当前状态、截止时间等 |
| **Inbox** | 当前用户作为接收者的任务列表 |
| **Outbox** | 当前用户作为派发者的任务列表 |

一条派发在云端通常对应 **1 条 Assignment + 接收端 1 条 Assigned Task 视图**（不必在 A 的 personal 列表里复制一份，除非产品需要「我派出的也显示在我的待办里」）。

## 4. 任务状态机

协作任务建议使用明确状态，避免与 personal 的 `done: bool` 混用：

```
                    ┌──────────┐
         创建派发   │ pending  │  待接收（接收者未处理）
        ──────────►│ (待确认) │
                    └────┬─────┘
           接受          │          拒绝
              ┌──────────┼──────────┐
              ▼                     ▼
       ┌────────────┐         ┌────────────┐
       │ accepted   │         │ rejected │
       │ (进行中)   │         │ (已拒绝) │
       └─────┬──────┘         └────────────┘
             │
    完成     │     撤回（仅派发者，且未 completed）
             ▼
       ┌────────────┐
       │ completed  │
       └────────────┘
             │
    撤回（派发者）
             ▼
       ┌────────────┐
       │ cancelled  │
       └────────────┘
```

| 状态 | 接收者可做 | 派发者可做 |
|------|------------|------------|
| `pending` | 接受 / 拒绝 | 撤回 |
| `accepted` | 标记完成、改备注（可选） | 查看进度、撤回（策略可选） |
| `rejected` | — | 重新派发（新建 Assignment） |
| `completed` | — | 查看 |
| `cancelled` | — | — |

## 5. 数据模型

### 5.1 云端表结构（PostgreSQL 示例）

**users**（可与 Auth 提供商用户表关联）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID | 主键 |
| email | text | 登录邮箱（唯一） |
| display_name | text | 显示名 |
| created_at | timestamptz | |

**task_assignments**（协作真相表）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID | 主键 |
| title | text | 任务标题 |
| description | text | 可选详情 |
| priority | smallint | 0/1/2，与现有 PRIORITY_* 一致 |
| due_at | timestamptz | 可选截止时间 |
| assigner_id | UUID | 派发者 |
| assignee_id | UUID | 接收者 |
| status | text | pending / accepted / rejected / completed / cancelled |
| assignee_note | text | 接收者备注（可选） |
| created_at | timestamptz | |
| updated_at | timestamptz | |
| completed_at | timestamptz | |

**assignment_events**（可选，审计与调试）

| 字段 | 类型 |
|------|------|
| id, assignment_id, actor_id, action, payload, created_at |

### 5.2 客户端扩展现有 `Task`

在实现 [跨平台账号同步方案](./跨平台账号同步方案.md) 中的 UUID 改造后，为协作任务增加字段：

```python
@dataclass
class Task:
    id: str                          # UUID
    text: str
    done: bool = False
    priority: int = PRIORITY_MED
    created: float = ...
    completed_at: float | None = None

    # 协作扩展
    kind: str = "personal"           # personal | assigned_in | assigned_out
    assignment_id: str | None = None
    assigner_id: str | None = None
    assigner_name: str | None = None
    assignee_id: str | None = None
    assignee_name: str | None = None
    assignment_status: str | None = None  # pending / accepted / ...
    due_at: float | None = None
    updated_at: float = ...
```

本地 `todos.json` 可拆成：

- `personal_tasks.json` — 仅本人待办  
- `assigned_inbox.json` — 派给我的（缓存）  
- `assigned_outbox.json` — 我派出的（缓存）  

或统一为一个文件，用 `kind` 区分（MVP 更简单）。

## 6. API 设计（REST 示例）

基础路径：`/api/v1`，所有请求需 `Authorization: Bearer <token>`。

### 6.1 用户目录

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/users/search?q=email前缀` | 按邮箱/昵称搜索可派发对象（需防枚举，见第 9 节） |
| GET | `/users/me` | 当前用户信息 |

### 6.2 任务派发

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/assignments` | 创建派发（body: assignee_id, title, priority, due_at, description） |
| GET | `/assignments/inbox` | 我收到的（支持 `?status=pending`） |
| GET | `/assignments/outbox` | 我派出的 |
| GET | `/assignments/{id}` | 详情 |
| PATCH | `/assignments/{id}` | 状态变更（见下表） |

**PATCH 动作示例**

| 调用方 | body | 结果 |
|--------|------|------|
| 接收者 | `{ "action": "accept" }` | status → accepted |
| 接收者 | `{ "action": "reject", "note": "..." }` | status → rejected |
| 接收者 | `{ "action": "complete" }` | status → completed |
| 派发者 | `{ "action": "cancel" }` | status → cancelled |

服务端校验：**只有 assignee 能 accept/reject/complete；只有 assigner 能 cancel**。

### 6.3 同步增量（与 inbox/outbox 配合）

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/assignments/changes?since=<unix>` | 返回当前用户相关的变更（inbox + outbox 合并） |

客户端启动或定时调用，合并到本地缓存并刷新 UI。

## 7. 典型业务流程

### 7.1 A 向 B 派发任务

1. A 在控制台「派发任务」输入 B 的邮箱（或从联系人选择）。
2. 客户端 `POST /assignments`。
3. 服务端写入 `task_assignments`，status = `pending`。
4. 向 B 发送通知（应用内 + 可选推送）。
5. A 的 Outbox 显示「待 B 确认」。

### 7.2 B 接受并执行

1. B 打开「派给我的」列表，看到 pending 任务。
2. B 点击「接受」→ `PATCH accept`。
3. 任务进入 B 的进行中列表；可与 personal 待办并列展示，或单独 Tab。
4. B 完成后点击完成 → `PATCH complete`。
5. A 的 Outbox 更新为「已完成」，并收到通知。

### 7.3 B 拒绝

1. B 点击「拒绝」并可选填写原因。
2. A 收到通知，Outbox 显示 rejected；A 可修改后重新派发（新 Assignment）。

### 7.4 离线场景

- 派发/接受/完成操作写入本地 **outbox 队列**，联网后按序重试。
- 以服务端状态为准；冲突时（如 A 已撤回，B 仍点接受）返回 409，客户端提示并刷新。

## 8. 客户端改造要点

### 8.1 模块划分（建议）

```
walle/
  sync/
    auth.py           # 登录（同步方案已有）
    sync_engine.py    # 个人数据同步
    assignment_api.py # 派发 API 封装
    assignment_store.py # inbox/outbox 本地缓存
  todo_manager.py     # 扩展：合并 personal + assigned 展示
```

### 8.2 UI（桌面控制台）

| 入口 | 功能 |
|------|------|
| 待办 Tab 子页 | 「我的待办」/「派给我的」/「我派出的」 |
| 派发按钮 | 选择接收者、标题、优先级、截止时间 |
| 任务卡片 | 显示派发者/接收者昵称、状态、截止日 |
| 账号页 | 登录状态、未读协作通知数 |

Android `mobile/` 需同样三个列表或 Tab；mobile 当前仅 `todo_store.py`，需先对齐桌面数据模型。

### 8.3 与现有 `TodoManager` 的衔接

- **方案 A（推荐）**：新建 `AssignmentManager`，与 `TodoManager` 并列；列表 UI 聚合两者。
- **方案 B**：将 `TodoManager` 泛化为 `TaskRepository`，内部按 `kind` 分支。

灯泡/宠物动画若绑定「待办数量」，需约定：是否包含 `pending` 的 inbox 任务（建议包含，提醒接收者处理）。

## 9. 权限、安全与隐私

| 风险 | 对策 |
|------|------|
| 任意向陌生人派发骚扰 | 仅允许向**已互相关注/同团队**用户派发，或需接收者「允许陌生人派发」开关 |
| 邮箱枚举 | 搜索接口限流、需完整邮箱匹配或双方已是联系人 |
| 越权改状态 | 服务端严格校验 assigner_id / assignee_id |
| 数据泄露 | 全程 HTTPS；Assignment 仅对相关用户可见（行级安全 RLS） |
| 推送内容 | 通知正文避免敏感详情，详情进 App 内查看 |

Supabase 可对 `task_assignments` 启用 RLS 示例策略：

- `SELECT`：`auth.uid() IN (assigner_id, assignee_id)`
- `INSERT`：`auth.uid() = assigner_id`
- `UPDATE`：assignee 可 accept/reject/complete；assigner 可 cancel

## 10. 通知

| 事件 | 派发者 | 接收者 |
|------|--------|--------|
| 新派发 | — | ✅ |
| 已接受 | ✅ | — |
| 已拒绝 | ✅ | — |
| 已完成 | ✅ | — |
| 已撤回 | ✅（若未读） | ✅ |

实现选项：

- **MVP**：客户端轮询 `/assignments/changes?since=` + 应用内角标。
- **V2**：WebSocket / Supabase Realtime / FCM（Android）推送。

桌面宠物可在收到新 inbox 时播放动画或托盘提示（与现有提醒体系复用 `ReminderManager` 的展示通道，但数据源来自协作 API）。

## 11. 后端选型（在同步方案之上）

| 方案 | 派发实现方式 |
|------|----------------|
| **Supabase** | PostgreSQL `task_assignments` + RLS + Edge Function 发邮件/推送 |
| **LeanCloud** | `_User` + 自定义 `Assignment` 类 + ACL 仅读写相关用户 |
| **自建 FastAPI** | 完全自定义 API + Celery 发通知 |

与个人同步共用同一 Auth 与用户表，避免两套账号体系。

## 12. 实施路线

| 阶段 | 交付物 | 依赖 |
|------|--------|------|
| **Phase 0** | 账号登录 + 个人待办 UUID 同步 | [跨平台账号同步方案](./跨平台账号同步方案.md) MVP |
| **Phase 1** | 云端 Assignment 表 + POST/GET inbox/outbox + 桌面「派给我的」只读列表 | Phase 0 |
| **Phase 2** | 接受/拒绝/完成/撤回 + Outbox + 基础通知（轮询） | Phase 1 |
| **Phase 3** | 派发 UI（搜用户、选优先级、截止日）+ Android 对齐 | Phase 2 |
| **Phase 4** | 推送、联系人/团队、离线队列、审计日志 | Phase 3 |

预估：在同步 MVP 完成后，协作 MVP 约 **2–3 周**（单人兼职开发）。

## 13. MVP 验收标准

- [ ] 用户 A、B 可分别登录不同账号  
- [ ] A 向 B 派发一条带优先级与截止时间的任务  
- [ ] B 在「派给我的」中看到 pending，可接受或拒绝  
- [ ] B 接受后可标记完成，A 在「我派出的」中看到 completed  
- [ ] A 可在 pending 阶段撤回  
- [ ] 桌面与 Android 至少一端完整跑通；另一端可只读 inbox  
- [ ] 未登录用户仍可使用纯本地 personal 待办（向后兼容）

## 14. 临时 / 轻量替代（无完整协作后端）

| 方式 | 说明 | 局限 |
|------|------|------|
| 共享链接 + 只读 | 生成任务链接，他人浏览器打开 | 非账号体系，难双向状态同步 |
| 共用团队账号 | 多人同账号登录 | 无个人边界，不推荐 |
| 导出 JSON 人工传递 | 邮件发送 todos 片段 | 无实时、无状态机 |

这些**不能**替代正式跨账号派发，仅作原型演示。

## 15. 小结

跨账号任务派发的实现路径：

> **账号体系（同步方案 Phase 0）→ 云端 Assignment 与状态机 → inbox/outbox API → 客户端缓存与 UI 三列表 → 通知与权限加固**

与现有代码的主要扩展点：

- `walle/todo_manager.py` — 数据模型与展示聚合  
- 新建 `walle/sync/assignment_*.py` — API 与本地 inbox/outbox  
- `walle/control_panel.py` — 派发与协作 Tab  
- `mobile/todo_store.py` — 与桌面模型对齐
